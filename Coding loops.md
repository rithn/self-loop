# Build-Verify Loop (`/code-build-loop`)

> Type `/code-build-loop` to generate project-specific builder and verifier agent prompts, the loop shell script, and a `scripts/CLAUDE.md` reference — then launch the build loop.

---

## What it does

Takes a project directory (with `prompts/app_spec.txt` and `prompts/tickets.md` already created by `/code-create-spec`) and sets up the autonomous build-verify pipeline:

1. **Asks for the project directory path** and reads the spec and tickets
2. **Confirms loop settings** — tickets per cycle, max retries, run name
3. **Pre-flight checks**:
   - Detects prior run logs and pre-populates completed tickets so the loop skips already-done work
   - Initialises git if the directory is not already a repo
   - Checks for `.env.required` and warns if `.env` is missing
4. **Generates five files**: `prompts/builder.md`, `prompts/verifier.md`, `scripts/run_build_verify_loop.sh`, `scripts/CLAUDE.md`, and optionally wires up a `scripts/.loop-cmd` for restarts
5. **Tells you the exact command to start the loop**

---

## Two-agent pipeline

Each cycle the shell script picks the next N tickets (configurable, default 2) and runs two agents in sequence:

1. **Builder** — reads the assigned tickets from `prompts/tickets.md` and `prompts/app_spec.txt`, implements each ticket, runs the exact verification commands specified in the ticket, and appends a result block to `build_report.md`. Outputs `BUILD DONE: COMPLETE / PARTIAL / FAILED`.

2. **Verifier** — independently re-runs the same verification commands without trusting the builder's self-reported results, tests corner cases beyond what the ticket specifies, reads the code for sanity, and checks `prompts/plan.md` (if present) for business rule compliance. If all tickets pass: commits with `git add -A && git commit`. If any fail: reports detailed failure notes for the builder to read on the next retry. Outputs `VERIFY DONE: PASS / FAIL`.

The verifier does not fix code — it diagnoses and reports. The builder handles fixes on the next cycle.

## Starting the loop

```bash
bash scripts/run_build_verify_loop.sh \
  --builder-prompt prompts/builder.md \
  --verifier-prompt prompts/verifier.md \
  --tickets prompts/tickets.md \
  --run-name <name> \
  --max-retries 3 \
  --tickets-per-session 2
```

Monitor live: `tmux attach -t agent-<name>`
Stop cleanly between cycles: `touch scripts/agent-run-logs/<name>/.done`

---

## Ticket progress states

Tracked in `scripts/agent-run-logs/<run-name>/.ticket_progress`:

| Status | Meaning |
|---|---|
| `COMPLETE` | Verified and committed |
| `IN_PROGRESS` | Currently being worked |
| `FAILED:N` | Failed N times, will retry up to max-retries |
| `INTERRUPTED` | Loop was stopped mid-cycle |
| `ABANDONED` | Exceeded max retries — see `abandoned_tickets.md` |

---

## Logs

All output goes to `scripts/agent-run-logs/<run-name>/`:
- `loop.log` — shell-level cycle log with timestamps
- `build_report.md` — running record of every builder and verifier result
- `.ticket_progress` — current state of each ticket
- `builder_cycle_N.log` / `verifier_cycle_N.log` — full agent output per cycle
- `abandoned_tickets.md` — tickets that exceeded max retries

---

## Post-run spec sync

After a run completes, the skill offers to update `app_spec.txt` with the actual build state — marking which tickets completed, which are pending, and what the implemented feature count is.

---

## Loop script features

The generated `run_build_verify_loop.sh` handles:
- **Env var validation** at startup — reads `.env.required`, sources `.env`, fails fast with a clear message if any required variable is missing
- **10-hour max runtime** safety cap
- **Graceful interruption** — Ctrl-C marks in-progress tickets as INTERRUPTED and saves state
- **tmux session management** with 50k-line scrollback
- Each agent runs in its own named tmux window so you can watch builder and verifier separately
